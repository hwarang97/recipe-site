## Project Overview

`오늘 뭐 먹지`는 단순히 레시피 사이트를 만드는 것보다, 직접 구현해보고 싶었던 기능을 실습해보기 위해 시작한 프로젝트입니다.

이 프로젝트에서 먼저 두고 싶었던 기술적 목표는 아래 두 가지였습니다.

- 이미지 데이터를 어떻게 저장하고 다룰지 설계해보기
- JWT 기반 로그인과 쿠키 인증 흐름을 직접 구현해보기

이 목표를 자연스럽게 담아낼 수 있는 주제로 레시피 공유 서비스를 선택했고, 그 과정에서 화면 구조 설계와 서버 렌더링 방식까지 함께 경험하는 것을 목표로 했습니다.

주요 학습 키워드는 다음과 같습니다.

- WireFrame
- SSR
- JWT
- Cookie
- 외부 스토리지(S3)

- 즉, MongoDB와 S3가 직접 연결되는 구조가 아니라 Flask 서버가 각 서비스를 각각 사용하도록 설계한 형태입니다.

## Authentication Flow

현재 로그인 흐름은 Flask 서버와 쿠키 기반 JWT 인증을 중심으로 구성되어 있습니다.

| 단계 | 설명 |
| --- | --- |
| 1 | 사용자가 로그인 요청을 보내면 Flask 서버가 입력한 계정 정보를 DB의 사용자 정보와 비교합니다. |
| 2 | 인증에 성공하면 `PyJWT` 라이브러리로 JWT를 생성합니다. |
| 3 | 생성된 JWT를 `HttpOnly` 쿠키로 발급합니다. |
| 4 | 이때 쿠키에 `max_age=60`을 설정해, 브라우저 기준으로 60초 뒤 쿠키가 만료되도록 했습니다. |
| 5 | 브라우저는 유효한 쿠키가 남아 있는 동안 이후 요청마다 해당 쿠키를 자동으로 전송합니다. |
| 6 | 서버는 화면 렌더링 과정에서 쿠키의 JWT를 확인하고, `jwt.decode()`를 통해 JWT 서명 검증을 수행합니다. |
| 7 | 검증에 성공하면 로그인 상태로 판단하고, 마이페이지처럼 인증 상태를 전제로 하는 화면 접근에 활용합니다. |

현재 코드 기준으로는 `PyJWT`를 사용해 JWT 생성과 JWT 서명 검증을 수행하고 있으며, 쿠키 발급도 확인할 수 있습니다. 로그인 유지 시간은 JWT의 `exp` 클레임이 아니라 쿠키의 `max_age=60` 설정으로 60초로 제한해두었습니다.

## 기술적 고민과 해결 방향

이 프로젝트에서 크게 고민했던 주제는 이미지 저장 방식과 JWT 로그인 구현 방식이었습니다.

### 1. 이미지를 어디에 저장할 것인가

처음에는 이미지를 MongoDB에 직접 저장하는 방법을 먼저 떠올렸습니다. 하지만 코치님들로부터 일반적인 선택은 아니라는 피드백을 받았고, 그 이유를 더 찾아보면서 외부 스토리지 사용 방향을 검토하게 됐습니다.

제가 정리한 핵심은 아래와 같습니다.

- 이미지처럼 용량이 큰 데이터를 DB 레코드에 직접 넣으면 조회와 저장 측면에서 부담이 커질 수 있습니다.
- 데이터베이스는 상대적으로 중요한 정형 데이터를 안정적으로 다루는 데 초점이 있고, 운영 비용도 더 높게 잡히는 경우가 많습니다.
- 반면 이미지 파일은 외부 스토리지에 두고, DB에는 이미지 URL이나 메타데이터만 저장하는 구조가 더 적절합니다.

이런 이유로 이미지 자체는 S3 같은 스토리지에 저장하고, MongoDB에는 이미지 위치 정보만 저장하는 방향이 맞다고 판단했습니다. 이후 코치님들께 다시 확인을 받았고, MongoDB와 S3를 함께 사용하는 구조로 생각을 정리했습니다.

### 2. JWT를 이용한 로그인을 어떻게 구현할 것인가

JWT는 `header`, `payload`, `signature`로 구성되고, 서버는 `signature`를 바탕으로 토큰이 위조되지 않았는지 검증합니다. 이 구조를 이해한 뒤 직접 생성 과정을 구현해보려 했지만, 인코딩 결과가 예상과 다르게 나오는 부분이 있었고 원인을 빠르게 확신하기 어려웠습니다.

그래서 실제 프로젝트에서는 `PyJWT` 라이브러리를 사용해 JWT 생성과 검증을 구현했습니다. 이 선택을 통해 직접 모든 세부 구현을 다시 쓰는 대신, 인증 흐름 자체를 먼저 안정적으로 동작시키는 데 집중할 수 있었습니다.

구현 과정에서 정리한 포인트는 아래와 같습니다.

- 로그인 성공 시 JWT를 생성한다.
- 생성한 토큰을 쿠키로 발급한다.
- 쿠키에 `max_age=60`을 설정해 로그인 유지 시간을 60초로 제한한다.
- 이후 요청에서 브라우저가 쿠키를 자동으로 전송한다.
- 서버는 전달된 토큰에 대해 JWT 서명 검증을 수행한다.
- 검증에 성공하면 로그인 상태로 판단하고 인증이 필요한 화면 접근에 활용한다.

브라우저 개발자 도구를 통해 쿠키가 실제로 발급되는 것도 확인했습니다. 현재 로그인 유지 시간은 쿠키 만료 시간으로 제어하고 있습니다.

## Image Storage

이 영역은 아직 구현이 완료되지 않았습니다.

- 현재 이미지 업로드 기능은 구현 예정입니다.
- 향후에는 사용자가 업로드한 이미지를 AWS S3에 저장하고,
- MongoDB에는 이미지 URL 또는 관련 메타데이터를 저장하는 구조로 확장할 계획입니다.

즉, 현재 저장소에는 이미지 처리 방향과 인프라 선택 의도는 반영되어 있지만, 실제 업로드 기능까지 완료된 상태는 아닙니다.

## Getting Started

배포 환경은 AWS EC2 / S3를 기준으로 구성했지만, 비용과 실행 환경 차이를 고려해 README에서는 로컬 실행 기준으로 안내합니다.

### 실행 순서

1. 레포지토리 클론
2. 가상환경 생성
3. 가상환경 활성화
4. 의존성 설치
5. 서버 실행
6. 브라우저 접속

### 1) 레포지토리 클론

```bash
git clone https://github.com/hwarang97/recipe-site.git
cd recipe-site
```

### 2) 가상환경 생성

```bash
python -m venv .venv
```

### 3) 가상환경 활성화

Windows

```bash
.venv\Scripts\activate
```

Mac / Linux

```bash
source .venv/bin/activate
```

### 4) 의존성 설치

```bash
pip install -r requirements.txt
```

### 5) 서버 실행

```bash
python app.py
```

### 6) 브라우저 접속

아래 주소 중 하나로 접속하면 됩니다.

```text
http://localhost:5000
http://127.0.0.1:5000
```

## 회고

이번 프로젝트에서는 팀원 간 기초 역량 차이가 예상보다 컸고, 그로 인해 인증, DB, 배포와 같은 구현 난도가 높은 작업이 제게 집중되었습니다. 당시에는 이 차이를 사전에 정확히 예상하기 어려웠기 때문에, 개발 과정에서 역할 분산보다 기능 완성이 우선되는 흐름으로 진행되었습니다.

이 경험을 통해, 협업에서는 단순히 역할을 나누는 것보다 초반에 작은 단위의 작업을 함께 수행해보며 각자의 이해도와 작업 속도를 먼저 확인하는 과정이 중요하다는 점을 배웠습니다. 다음에는 초기 단계에서 팀의 현재 수준을 더 빠르게 파악하고, 그에 맞춰 범위와 역할을 조정하는 방식으로 협업을 운영하고 싶습니다.
